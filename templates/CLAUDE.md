# <PROJECT_NAME> Claude Code Operating Guide

> Jamong 전용 Claude Code 프로젝트 운영 템플릿입니다. 이 파일을 대상 프로젝트 루트의 `CLAUDE.md`로 복사한 뒤 `<PROJECT_NAME>`과 `<PROJECT_ROOT>`를 실제 값으로 바꿔 사용합니다.

## 0. 우선순위

규칙 충돌 시 아래 순서를 따릅니다.

1. 사용자의 최신 명시 요청
2. 프로젝트 루트의 `CLAUDE.md` / `AGENTS.md` / `.cursorrules`
3. 저장소 문서: `README.md`, `docs/*`, `CHANGELOG.md`
4. 글로벌 Jamong 운영 규칙
5. 일반적인 프레임워크/언어 관례

명시 요청이 없는 한 commit, tag, push, release, deploy, 서비스 재시작 등 외부 상태 변경은 수행하지 않습니다.

## 1. 커뮤니케이션 규칙

- 기본 응답 언어는 한국어입니다.
- 진행 상황은 짧고 명확하게 공유합니다.
- 불확실한 사실은 추측하지 않고 확인합니다.
- 필요한 컨텍스트가 파일/명령으로 확인 가능하면 먼저 확인합니다.
- 단순한 선택지는 합리적인 기본값으로 진행하고, 결과에 영향이 큰 모호성만 질문합니다.
- 완료 보고에는 변경 파일, 검증 결과, 남은 리스크, Claude/Codex 사용량 확인 가능 여부를 포함합니다.

## 2. 작업 디렉터리 규칙

- Jamong의 프로젝트는 기본적으로 `/home/dev/project/<project>` 아래에 둡니다.
- 현재 프로젝트 루트는 `<PROJECT_ROOT>` 입니다.
- 명령 실행 전 현재 위치와 대상 파일을 확인합니다.
- 다른 프로젝트나 홈 디렉터리의 전역 설정을 변경해야 하는 경우, 사용자가 명시적으로 요청했는지 확인합니다.

## 3. 관리자 권한 금지

AI 에이전트는 `sudo`, `doas`, `pkexec` 등 관리자 권한 명령을 직접 실행하지 않습니다.

관리자 권한이 필요한 경우 즉시 중지하고 아래 문구를 정확히 출력한 뒤, 사용자가 직접 실행할 명령을 제공합니다.

```text
관리자 권한이 필요한 작업이라 중지했습니다, 아래 명령어를 실행 후 이어서 진행해주세요
```

예시:

```text
관리자 권한이 필요한 작업이라 중지했습니다, 아래 명령어를 실행 후 이어서 진행해주세요

sudo apt install <package>
```

## 4. Git / 배포 안전 규칙

명시 요청 없이 수행 금지:

- `git commit`
- `git tag`
- `git push`
- PR 생성/병합
- GitHub Release 생성
- 배포 명령
- 서비스 restart/reload
- 원격 리소스 삭제/변경

허용되는 기본 Git 작업:

- `git status`
- `git diff`
- `git log` 조회
- 브랜치/원격 정보 조회

릴리즈/배포가 명시된 경우:

- CHANGELOG를 먼저 갱신합니다.
- 태그 규칙은 `YY.major.minor` 또는 `YY.MM.N`을 따릅니다.
- 기본 배포 방식은 GitHub Releases + GitHub Actions 입니다.
- 서비스 재시작이 필요하면 재시작 후 정상 동작을 검증합니다.

## 5. 개발 원칙

### Think Before Coding

- 요구사항과 성공 기준을 먼저 정리합니다.
- 중요한 가정은 명시합니다.
- 자료가 부족하면 파일/문서/명령으로 확인합니다.

### Simplicity First

- 가장 작은 해결책을 우선합니다.
- 요청 범위를 벗어난 리팩터링, 추상화, 대규모 구조 변경을 피합니다.
- 새 의존성 추가는 명확한 이득이 있을 때만 합니다.

### Surgical Changes

- 작업과 직접 관련된 파일/라인만 수정합니다.
- 포맷터로 인한 대량 변경을 피합니다.
- 기존 스타일과 네이밍을 존중합니다.

### Goal-Driven Execution

- 변경 후 성공 기준을 검증합니다.
- 실패한 검증은 숨기지 않고 원인과 다음 조치를 보고합니다.

## 6. 코드 변경 후 검증 체크리스트

코드/설정/문서 변경 후 가능한 범위에서 아래를 수행합니다.

1. 변경 확인
   - `git diff -- <files>`
   - 의도하지 않은 파일 변경 여부 확인
2. 테스트
   - 프로젝트에 맞는 테스트 명령 실행
   - 예: `npm test`, `pnpm test`, `pytest`, `go test ./...`, `cargo test`
3. 린트/타입체크
   - 예: `npm run lint`, `pnpm lint`, `ruff check`, `mypy`, `tsc --noEmit`
4. 보안/시크릿 점검
   - `.env`, API key, token, private key가 커밋 대상에 포함되지 않았는지 확인
   - 가능한 경우 보안 스캔 실행
5. 문서 갱신
   - 동작/명령/설정이 바뀌면 README 또는 docs 갱신
   - 릴리즈/배포 요청이 있으면 CHANGELOG 갱신

검증 명령이 없거나 실패하면 완료 보고에 `미실행/실패 사유`를 명시합니다.

## 7. Claude Code 운영 규칙

- Claude Code는 개발 작업의 기본 에이전트로 우선 고려합니다.
- 단발성 작업은 가능하면 print mode를 사용합니다.
- 긴 작업은 tmux 또는 백그라운드 프로세스로 추적 가능하게 실행합니다.
- `--max-turns` 등 제한을 설정해 runaway를 방지합니다.
- 위험한 권한 우회 옵션은 사용자가 명시적으로 요구한 경우에만 고려합니다.
- 5시간 사용량이 70% 근처면 사용자에게 경고합니다.
- 5시간 사용량이 90%를 넘거나 사용이 어려우면 Codex/OMX 대안을 고려합니다.
- `/compact`는 사용자가 요청하거나 작업 지속에 꼭 필요한 경우에만 수행합니다.

## 8. Codex / OMX 운영 규칙

- Jamong은 Codex를 직접 실행하기보다 보통 아래 명령으로 OMX를 통해 실행합니다.

```bash
omx --madmax --high
```

- Codex/OMX는 git 저장소 안에서 실행합니다.
- 장시간 작업은 백그라운드/세션 추적 방식으로 실행하고 로그를 확인합니다.
- Claude Code 사용량이 높거나 특정 작업에 Codex가 더 적합할 때 대안으로 사용합니다.
- Codex 사용량을 비대화형으로 신뢰성 있게 확인할 수 없으면 `확인불가`로 보고합니다.

## 9. RTK / 토큰 절약 규칙

- RTK가 설치되어 있으면 큰 출력 명령에는 RTK 사용을 우선 고려합니다.
- Claude/Codex의 Bash hook에서는 관리자 권한 차단이 RTK rewrite보다 먼저 실행되어야 합니다.
- 내장 Read/Grep/Glob 도구는 Bash hook/RTK를 거치지 않을 수 있으므로 큰 출력이 예상되면 shell 기반 `rtk` 명령을 고려합니다.

권장 hook 순서:

1. 도구/플러그인 정책 hook
2. Jamong 관리자 권한 차단 hook
3. RTK rewrite hook

## 10. 파일 작성/수정 규칙

- 기존 파일은 먼저 읽고 구조를 파악한 뒤 수정합니다.
- 새 파일은 목적, 사용법, 유지보수 포인트가 드러나게 작성합니다.
- 템플릿 파일은 실제 프로젝트에 복사해도 바로 이해될 수 있도록 설명을 포함합니다.
- 시크릿/토큰/개인정보는 작성하지 않습니다.
- `.env`류 파일은 예시가 필요하면 `.env.example`로 작성합니다.
